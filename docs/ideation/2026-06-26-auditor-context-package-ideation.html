Grounding Context

What already exists. OVRFLO is 1161 nSLOC across three subsystems — core vault (OVRFLO.sol wrap/unwrap reserve + PT deposit/claim + Sablier stream creation, OVRFLOToken.sol), the timelocked-multisig-owned admin hub (OVRFLOFactory.sol), and the secondary market (OVRFLOBook.sol + StreamPricing.sol). The repo is already unusually audit-prepared: a protocol-spec README.md, a domain glossary CONCEPTS.md, security guideline docs, and a full x-ray/ suite: x-ray.md (overview + threat/trust model), entry-points.md (37 entry points), invariants.md (19 guards + 9 single-contract + 5 cross-contract + 3 economic invariants), architecture.svg, and a 10-persona multi-agent-audit-report.md. There are 16 test files / 161 test functions including 3 stateful-fuzz wrap/unwrap harnesses and mainnet-fork suites.

The gaps the ideation targets. Despite all of that, the material the user explicitly asked about — the external-protocol mental models — is the thinnest. There is no dedicated, dependency-scoped explainer of Pendle (PT/SY, maturity convergence, getPtToSyRate TWAP, observation cardinality, what Pendle governance can mutate) or Sablier V2 (stream-NFT sender/recipient/owner semantics, withdrawableAmountOf, non-cancelable + no-cliff, the v1.1 withdraw ACL). External-dependency trust assumptions are scattered across the x-ray, invariants X-2/X-5, and the audit report; there is no single index orienting an auditor to the corpus; and reproduction (fork tests, coverage) is blocked behind unstated MAINNET_RPC_URL / dependency-address setup. The four invariants flagged not enforced on-chain (I-2, I-7, X-2, X-5) are precisely where residual risk lives, but they are buried.

External best-practice basis. Audit-readiness guidance (Smart Contract Security Field Guide, plus Hacken / Quantstamp / Sherlock readiness checklists) converges on the same package: a pinned scope doc (branch + commit + in-scope paths), a domain glossary, a threat model with access controls, explicit known-issues / risk disclosure, defined invariants, and reproducible build/test.

Topic Axes

  1. External-protocol mental models — Pendle + Sablier internals OVRFLO depends on, scoped to OVRFLO's actual usage and trust assumptions.
  2. Internal protocol model & economic design — OVRFLO flows, value split, dual-backing accounting, self-repaying-loan economics.
  3. Security-reasoning scaffolding — threat model, invariants, trust boundaries, known/rejected findings, what to attack.
  4. Verifiability & reproduction — runnable tests/coverage/fork, pinned commit + dependency addresses, executable invariant/property harnesses.
  5. Navigation & packaging / delivery — how the corpus is indexed, sequenced, versioned, and handed off.

I1

Dependency Interface Contract (Pendle + Sablier)

Axis 1 · External mental models Confidence 85% Complexity: Medium Top pick

One artifact per external dependency, scoped to exactly what OVRFLO calls, framed not as a tutorial but as an interface contract the auditor tries to falsify: each external call (getPtToSyRate, createWithDurations, withdrawableAmountOf, transferFrom, yieldToken) listed with the precondition OVRFLO assumes, the postcondition it relies on, where that is validated locally (or not, with file:line), and the failure mode if the assumption breaks. Each card is pinned to the deployed version (Sablier V2 Lockup Linear v1.1 at 0xAFb979…) and ships with a short runnable test that demonstrates the assumption against the real bytecode.

Description
Scoping to OVRFLO's actual usage means an auditor reads ~3 pages per dependency instead of Pendle's or Sablier's full docs, and the runnable demo (e.g. prank a stranger → SablierV2Lockup_Unauthorized) lets them confirm semantics against the deployed contract rather than trust prose.
Axis
1 — External-protocol mental models
Basis
direct: Grounding gaps #1/#2 (no scoped Pendle/Sablier explainer); invariants.md X-2 callee side ("oracle freshness/cardinality is only pre-checked at onboarding… not revalidated inside deposit path"); the verified Sablier v1.1 withdraw-ACL table in multi-agent-audit-report.md ("requires msg.sender to be stream sender, NFT owner, or approved operator… that does not apply to the V2 v1.1 bytecode" distinction that flipped H-2). external: NASA Interface Control Document practice; scsfg.io audit-prep "meaningful dependencies" requirement.
Rationale
The #1 ranked adversary in the x-ray is the oracle manipulator and the entire self-repaying-loan close path rests on Sablier semantics — both are external. An auditor who misreads the Sablier version (the exact thing that turned a "High" into a rejected finding internally) wastes the engagement re-deriving it or re-raises a false positive.
Downsides
Must be re-pinned whenever a dependency version/address changes, or it silently misleads. Requires care to scope tightly — the temptation to drift into a general Pendle tutorial defeats the purpose.
I2

Guided AUDIT.md index + citation graph + one-screen triage map

Axis 5 · Navigation & packaging Confidence 88% Complexity: Low Cheapest multiplier

A single root AUDIT.md that turns the already-rich-but-scattered corpus into a guided path: a prescribed reading order (scope/commit snapshot → dependency contracts → internal model → invariants, starting with the four not-enforced → known/rejected findings → reproduce), every artifact addressable by the stable IDs that already exist (G-/I-/X-/E- codes, entry-point names), a scope-exclusion log stating what is out of scope and why (e.g. "Sablier internals = bounded external, trusted at v1.1"), and a one-screen triage map: 37 entry points × invariant IDs × adversary ranking, flagged by reentrancy-guard presence and by the not-enforced set.

Description
The artifacts mostly exist; what is missing is the map. An index that sequences them and gives every item a citation handle converts a pile of docs into an addressable graph an auditor can navigate and reference findings against ("re: X-5 closeLoan / Sablier withdrawability").
Axis
5 — Navigation & packaging / delivery
Basis
direct: Grounding gap #4 (no single entry/index); entry-points.md already enumerates 37 entry points with per-function reentrancy flags; invariants.md already carries stable G-/I-/X-/E- IDs with file:line. external: scsfg.io "documentation as cornerstone" (scope doc + glossary + threat model); legal-discovery Bates indexing + privilege log (addressable corpus + documented exclusions).
Rationale
This is the highest leverage-to-cost idea: it multiplies the value of every artifact already written, removes the "where do I start / is this current?" tax, and the triage map ensures the auditor's first hour targets oracle and Book-settlement risk rather than admin boilerplate.
Downsides
An index that drifts from the artifacts it points to is worse than none; needs to be regenerated or checked at each audited commit. The one-screen map risks over-compression if it tries to encode too much.
I3

Reproducible audit harness (pinned + one-command + invariants-as-properties)

Axis 4 · Verifiability & reproduction Confidence 82% Complexity: High

Reframe the handoff from "document bundle" to "forge project the auditor clones and runs green before reading prose." It bundles an immutable scope snapshot (commit 8c603d8, in-scope/excluded paths, code-freeze declaration, deployed Sablier + Pendle market/oracle addresses, lib/ submodule commits, pinned fork block), a one-command environment that resolves the MAINNET_RPC_URL friction currently blocking coverage, and every entry in invariants.md compiled to a named runnable Foundry/Halmos property — with the four not-enforced invariants (I-2, I-7, X-2, X-5) shipped as deliberately documented-gap / red harnesses that show the off-chain-trusted boundary empirically.

Description
Prose invariants are unfalsifiable and rot; executable ones force precision, let the auditor probe boundaries in minutes, and double as a regression asset reused across every future market and re-audit. The pinned snapshot makes every finding reproducible rather than contestable ("that was fixed yesterday").
Axis
4 — Verifiability & reproduction
Basis
direct: x-ray Test Analysis — coverage "Unavailable — forge coverage run failed due missing MAINNET_RPC_URL", "Stateless Fuzz 0", no formal verification; audit-report test-gap "P1 — No invariant/fuzz on loan lifecycle interleaving"; invariants.md already carries on-chain Yes/No status + file:line for every invariant (the metadata needed to generate one property each). external: ACM artifact-evaluation / reproducibility-package practice; Sherlock/Quantstamp "defined invariants" as runnable artifacts.
Rationale
The single biggest reproduction friction today is the fork RPC env; removing it up front means zero setup time and immediate trust in the baseline. Encoding the not-enforced invariants as explicit harnesses converts the audit's highest-value targets from a buried footnote into observable evidence and closes the protocol's biggest current testing gap (loan-lifecycle interleaving).
Downsides
Real test-engineering cost — this is the heaviest survivor. Halmos/formal tooling has a learning curve; over-investing here could delay the cheaper, higher-ramp-value artifacts (I1, I2).
I4

Trust Assumption Ledger / "not-enforced-on-chain" pre-flight briefing

Axis 3 · Security-reasoning scaffolding Confidence 84% Complexity: Low

One surface that consolidates every belief OVRFLO asks the auditor to validate: the four invariants flagged On-chain: No (I-2 deposit-limit lowering, I-7 APR-drift on active orders, X-2 oracle freshness, X-5 Sablier withdrawability drift), every "bounded external" actor, and every documented trust boundary — each formatted as a line item the auditor explicitly marks ACCEPT or CHALLENGE. Framed as an aviation-style pre-flight briefing whose "NOTAM" section declares these known soft spots as hazards to probe, not surprises to discover.

Description
Off-chain-trusted invariants are exactly where real findings hide. Surfacing them as the audit's front door — and as explicit accept/challenge assertions — concentrates finite hours on genuine residual risk instead of re-confirming the 15 enforced guards, and pre-empts the "we assumed Pendle-only tokens" class of late-stage disagreement.
Axis
3 — Security-reasoning scaffolding
Basis
direct: invariants.md header "4 not enforced on-chain"; I-2/I-7/X-2/X-5 each carry On-chain: No with explicit "If violated" notes (e.g. I-7: "active orders can remain executable while outside the latest APR policy"); x-ray trust boundaries + actors/adversary ranking. external: FAA pre-flight briefing / NOTAM practice (declare known hazards to fly around).
Rationale
Most protocol/auditor disputes are about unstated assumptions; making them explicit and acknowledge-able is cheap and high-signal. It is also a trust move — publishing your own known soft spots up front reads as a mature team and redirects scrutiny where it pays.
Downsides
Requires a judgment call on how much to "lead the witness" — over-framing could anchor an auditor away from a real issue you mis-categorized as accepted. Needs sign-off from the team on what is genuinely accepted vs simply unaddressed.
I5

Known / rejected findings decision record + pre-answered Q&A bank

Axis 3 · Security-reasoning scaffolding Confidence 80% Complexity: Low

Promote the internal 10-persona review's settled conclusions into a persistent, citable decision record so the external auditor starts where the last review ended: the rejected/downgraded findings (H-2 Sablier withdraw-grief, rejected via the verified v1.1 ACL table keyed to 0xAFb979…; M-5 cross-market fungibility, by-design; H-1 uint128 narrowing downgraded to L-1), each with the evidence that closed it, plus a pre-answered Q&A bank seeded from the five "Resolved open questions" (Sablier withdraw ACL, permissionless closeLoan intent, Pendle-only token assumptions, multi-market same-expiry, stale listing fee).

Description
Auditors predictably rediscover and re-raise the same "obvious" items in week one. A rigorous prior-reasoning ledger converts a one-time (expensive) verification into permanent capital each subsequent audit extends rather than repeats, and signals a mature team.
Axis
3 — Security-reasoning scaffolding
Basis
direct: multi-agent-audit-report.md "Rejected findings" (H-2, M-5 with full reasoning), "Resolved open questions" table (5 Q/A), and "Reviewed — no medium+ issue" consensus table; the verified Sablier v1.1 ACL table tied to the deployed address. external: scsfg.io / Hacken require a known-issues / risk-disclosure deliverable.
Rationale
The exact Sablier-v1.1-vs-later-version distinction already cost a verification pass; without disclosure every new auditor re-derives it (or gets it wrong, as the initial internal pass did). Front-loading it redirects budget toward genuinely novel surface.
Downsides
Risk of leading the witness or appearing defensive; a careless reader might treat "rejected" as "don't look here." Must be framed as evidence to challenge, not conclusions to accept. Overlaps partly with I4 — the two should cross-link rather than duplicate.
I6

Dual-backing solvency tie-out + self-repaying-loan economics explainer

Axis 2 · Internal model & economics Confidence 76% Complexity: Medium

Explain OVRFLO's two subtlest internal mechanics as auditor-checkable accounting identities rather than happy-path prose. First: one fungible ovrfloToken is backed by two separately-accounted pools (matured PT claims via marketTotalDeposited vs underlying wrap reserve via wrappedUnderlying); present it as a balance sheet the auditor can "tie out" against on-chain state, framed insolvency-first ("here is every way the books could stop balancing"). Second: the self-repaying loan departs from auditor priors — no health checks, no loan-time oracle, no liquidations — so explain outstanding = obligation − (drawn + repaid), why permissionless closeLoan() is liveness not exploit, and the deposited − withdrawn pricing-at-fill rule.

Description
An auditor anchored to standard ERC-4626 / lending mental models will either flag missing liquidation logic as a finding (false positive) or miss the dual-backing seam (false negative). An insolvency-first, tie-out framing puts attention exactly where conservation invariants I-1/I-3/E-1/E-2/E-3 can break.
Axis
2 — Internal protocol model & economic design
Basis
direct: CONCEPTS.md ("two backing sources for the same receipt token… must remain separately accounted"; self-repaying loan "non-cancelable… cannot underperform"); economic invariants E-1/E-2/E-3 and conservation I-1/I-3; audit-report I-3 ("Permissionless closeLoan()… Accept as intentional liveness"); the deposited − withdrawn pricing note. external: double-entry / financial-audit tie-out methodology.
Rationale
This is the core of "understand how OVRFLO works" — its economic identity, not its function signatures. Framing it as a balance sheet gives the auditor an independent, on-chain-checkable solvency argument and pre-empts a whole class of prior-anchored false positives around the loan design.
Downsides
Lower confidence because some of this is already in CONCEPTS.md and the README's Design Notes — the win is consolidation + the insolvency-first reframe + the diagram, so it risks feeling like a repackage if not paired with the tie-out harness (links to I3).
I7

Annotated golden-path lifecycle traces / interactive walkthrough

Axis 1 · External mental models Confidence 72% Complexity: Medium

Capture the three core journeys (deposit→stream, list→take, borrow→claim→close) as committed, human-annotated fork-test traces that double as auditor walkthroughs and regression fixtures — each step labeled with the state it mutates, the guard it passes, and crucially who owns the Sablier NFT at that moment. Optionally drive it as a guided sandbox on the existing local-seed infrastructure (script/seed-local.sh / bootstrap:local) so a reviewer who has never seen Pendle or Sablier learns the dynamic interplay by running it.

Description
The hardest thing to transfer is the dynamic interaction between Pendle valuation and Sablier streaming — who can call withdraw at each lifecycle stage, when the oracle split is computed, how NFT ownership moves through the book. A static entry-point table doesn't expose the timing interleaving the #2 adversary (stream/MEV trader) probes.
Axis
1 — External-protocol mental models (the Pendle×Sablier interplay)
Basis
direct: fork suites already exist (test/fork/OVRFLOBookMainnetFork.t.sol) and the x-ray already documents these exact call chains step-by-step; script/seed-local.sh + bootstrap:local infra exists to build on; entry-points.md marks closeLoan() "Anyone" with the Sablier withdraw call chain. external: game tutorial-level / sandbox onboarding design.
Rationale
A trace is simultaneously the fastest way to transfer the dynamic model, a regression baseline, and an internal-onboarding asset — it earns its keep three ways and stays valid as long as it re-generates against the pinned commit.
Downsides
Lowest confidence: annotated traces drift if not regenerated, and a guided sandbox is build effort that overlaps the reproducible harness (I3). Risk of gilding — for a single audit, narrated walkthroughs of just the two costliest flows (deposit, closeLoan) may capture most of the value at a fraction of the cost.
