Grounding Context

Plan under evaluation. docs/plans/2026-07-18-002-feat-web-markets-rebuild-plan.md specifies a greenfield web/ rebuild on Next.js 16.2 (static export) + wagmi 3.7 + viem 2.55 + TanStack Query 5 + Tailwind 4 + Reown AppKit + Vitest 4 + @wagmi/cli foundry plugin for ABI typegen, with the Envio indexer pinned at 2.32.x and the existing CSP / static-export-verify / banned-patterns / ModalErrorBoundary infra carried over.

Current installed stack (web/package.json, verified this run): Next ^16.1.6, wagmi ^2.15.4 (v2, not v3), viem ^2.31.3, @reown/appkit ^1.7.3, @tanstack/react-query ^5.72.2, Tailwind ^4.1.4, Vitest ^4.0.18, TypeScript ^5.8.3. The app uses output: "export" with scripts/build-csp.mjs, scripts/verify-static-export.mjs, scripts/check-banned-patterns.sh, a class-component ModalErrorBoundary, and env-validation Vitest tests. The stale calculateMinFeeWei entry in web/lib/contracts.ts:153 (cited by the plan as the drift that broke the old app) is confirmed present.

Institutional learnings (docs/solutions/): pattern #1 (ALWAYS REQUIRED) — Envio GraphQL indexer is the sanctioned Sablier stream discovery path; log-scan fallback is banned by check-banned-patterns.sh. Pattern #2 (ALWAYS REQUIRED) — no forge script --broadcast on local Anvil forks. Pattern #3 (ALWAYS REQUIRED) — owned ModalErrorBoundary with onReset remount contract; react-error-boundary explicitly rejected. Patterns #18/#19 — external ABIs are documentation, not ground truth; verify with cast call. The five Web UI writeups (2026-04-21) document Next 15 / wagmi 2.x shapes; the architectural rules survive the v3 migration, the code snippets do not.

External research (web, 2026-07-18): for no-SSR/no-SEO wallet dapps, Vite + React SPA is the 2026 default recommendation; Next.js static export is common but carries framework weight for file-based routing only. wagmi v3 shipped Nov 2025 with connectors moved to peer deps; RainbowKit has no v3 support (discussion rainbow-me/rainbowkit#2575 unanswered); Reown AppKit is the healthiest-maintained modal but its wagmi v3 peer range was NOT explicitly confirmed in any fetched source. @wagmi/cli foundry plugin is the documented ABI typegen path. Tailwind 4 @theme compiles tokens to native CSS custom properties, collapsing the Tailwind-vs-vanilla-CSS dichotomy. E2E pattern is two-tier: Vitest + mock connector for logic, Playwright + Anvil + injected mock EIP-1193 for release gating (rombrom.com pattern); Synpress is the heavier alternative.

Sources: plan file; web/package.json; docs/solutions/patterns/ovrflo-critical-patterns.md; 5 Web UI / LocalSeeding writeups; web research digest (wagmi.sh, rainbow-me/rainbowkit#2575, designrevision.com, patterns.dev, aave/interface, synpress.io, rombrom.com, tailwindcss.com/blog, docs.envio.dev).

Stack Spike 0: verify wagmi v3 + Reown AppKit v3 peer compatibility

Conf HighCx Low
DescriptionBefore scaffolding any hooks, stand up a scratch branch that installs wagmi@^3.7, @reown/appkit@latest, @reown/appkit-adapter-wagmi@latest, and the v3-mandated connector peer deps (MetaMask SDK, WalletConnect, Coinbase). Render createAppKit, connect an injected wallet, and confirm useConnection/useWriteContract compile and fire. Only on green does the stack decision lock.
Axiswallet / connection
BasisexternalWeb research: wagmi v3 shipped Nov 2025 with connectors moved to peer deps (wevm/wagmi PR #4857); RainbowKit has no v3 support (rainbow-me/rainbowkit#2575 unanswered); Reown AppKit is actively released (v1.8.21, May 2026) but "verify its wagmi v3 peer range in package.json before committing; none of the fetched sources explicitly confirmed v3 compatibility." directweb/package.json currently pins @reown/appkit ^1.7.3 against wagmi ^2.15.4 — the v3 combination has never run in this repo.
RationaleKTD2 asserts "Reown AppKit (the only modal whose peer range admits wagmi v3)" as a settled fact, but the evidence base for that assertion is negative (RainbowKit is out) not positive (AppKit v3 is in). The plan's existing "first implementation step of the data layer" spike covers the wagmi rename surface, not AppKit compatibility. A failed spike is the cheapest possible signal that the stack needs a contingency (Survivor 5, or defer to wagmi v2) before the rebuild is underway.
DownsidesAdds a half-day of spike work before P2. If the spike fails, the plan needs a KTD2 amendment and a fallback decision before scaffolding can start — but that is exactly the failure mode the spike exists to surface early.
2

Keep Envio indexer on pinned 2.32.x — do not switch to Ponder or viem getLogs

Conf HighCx Low
DescriptionAffirm the plan's P6 choice: extend the vendored Envio indexer on V2 conventions. Resist the temptation to swap to Ponder (simpler ops) or plain viem getLogs (zero infra) that external research surfaces as mainstream 2026 alternatives. Carry the existing GraphQL payload-shape contract test (web/tests/lib/sablier.test.ts) forward verbatim.
Axisdata / caching
BasisdirectPattern #1 (ALWAYS REQUIRED) in docs/solutions/patterns/ovrflo-critical-patterns.md: "ERC-721 current ownership comes from the token/indexer, never from derived protocol events." web/scripts/check-banned-patterns.sh:32-33 greps to ban watchContractEvent.*Deposited and getLogs.*Deposited. The transferred-Sablier-NFTs writeup (high severity) documents three failed log-scan approaches and calls dual scanner+indexer coexistence "a maintenance trap." externalWeb research lists Ponder and viem getLogs as Envio alternatives but notes Envio is mainstream for self-hosted small protocols.
RationaleThe highest-severity frontend learning in this repo is that log-scan stream discovery is wrong and the Envio GraphQL indexer is the sanctioned path. Switching indexers throws away the contract test that enforces pattern #1 and reopens a closed high-severity bug. The external alternatives are mainstream in the abstract but do not carry this repo's institutional scar tissue.
DownsidesEnvio carries a Docker/Postgres ops dependency and a pinned 2.32.x version (V3 migration is explicitly deferred). That cost is already paid and already documented; switching would trade a known cost for an unknown one.
3

Add a drift test for the hand-maintained Sablier ABI against the vendored envio ABI

Conf HighCx Low
DescriptionKTD3 makes the Sablier Lockup ABI permanently hand-maintained (the repo's ISablierV2LockupLinear.sol is a narrow interface; frontend-only functions like withdrawMax, getRecipient, ERC-721 approvals have no forge artifact). Add a Vitest test that asserts every function signature in the hand-written sablierLockupAbi matches the corresponding entry in tools/envio/abi/SablierV2LockupLinear.json (and/or a Sourcify fetch). Generate from neither; just fail CI on drift.
Axisabi-pipeline
BasisdirectKTD3 specifies a CI drift test for the @wagmi/cli-generated OVRFLO ABIs ("a Vitest test asserting expected function/error names exist in the generated ABI") but explicitly carves out the Sablier ABI from generation. The stale calculateMinFeeWei at web/lib/contracts.ts:153 is the exact failure mode a drift test would have caught. Patterns #18/#19: external ABIs are documentation, not ground truth; verify empirically with cast call. directtools/envio/abi/SablierV2LockupLinear.json already exists in-repo as a verified source the plan cites.
RationaleThe @wagmi/cli plugin fixes drift for OVRFLO contracts structurally. The hand-maintained Sablier ABI is the one remaining drift surface, and it is the surface that already broke the old app (calculateMinFeeWei). A signature-equality test against the envio ABI closes that gap without the risk of pulling in unverified entries that full generation would introduce.
DownsidesThe envio ABI may itself drift from the deployed Sablier v1.1.2; the test should pin to the vendored JSON copy, not a live fetch. Adds one small Vitest file. Negligible cost.
4

Consider Vite + React SPA as the framework shape instead of Next 16 static export

Conf MediumCx Medium
DescriptionA genuine alternative to surface, not a switch to make blindly. Under Vite + React 19 the app is a single-page dashboard (the current web/app/ has one route plus modals), CSP ships as a static _headers entry instead of a build script, and the Next framework weight (RSC disabled by output: "export", headers() a no-op, route handlers unused) is removed. The ModalErrorBoundary, banned-patterns grep, env-validation tests, and DESIGN.md tokens are all framework-agnostic and carry over unchanged.
Axisframework / rendering
BasisexternalWeb research (designrevision.com Jul 2026, patterns.dev "React 2026"): "for a wallet-gated, no-SEO, no-SSR app, Vite + React SPA is the default recommendation; Next.js earns its keep only with SSR/RSC/route handlers — all of which static export disables." Aave uses Next; Uniswap runs a React monorepo (RN-web-driven, not applicable). directweb/next.config.ts:9 sets output: "export"; web/scripts/build-csp.mjs:8 comments "Next's headers() API is a no-op under output: 'export'" — the CSP script exists solely to work around that no-op. The app has one page (web/app/page.tsx) plus modals.
RationaleThe plan chose Next 16 because the team already has Next infra and familiarity. That is a real factor, but the rebuild is greenfield (KTD1), so the effort is comparable either way. The CSP scripts are the strongest Next lock-in signal, and they exist because of a Next limitation, not a Next feature. Worth a deliberate "did we pick Next by default or on purpose?" conversation before the rebuild starts.
DownsidesLoses the team's Next muscle memory, next/font (Inter + IBM Plex Mono load becomes a Vite font setup), and next/image (unused on a dark data UI). The CSP scripts need rewriting as static headers — simpler, but new code. TanStack Router would be overkill for one page; a minimal react-router or hand-rolled view switch suffices. This is a "consider seriously" survivor, not a "switch" recommendation.
5

Evaluate a custom DESIGN.md-compliant connect modal as a fallback if AppKit theming fights back

Conf MediumCx Medium
DescriptionIf Spike 0 (Survivor 1) passes but AppKit's default modal proves hard to bend to "Architectural Dark" (sharp corners, no shadows, obsidian/carbon/graphite, mono, 1px borders), have a fallback design: wagmi's useConnect + injected + WalletConnectConnector directly, with a ~200-line custom modal that matches DESIGN.md for free. Sidesteps the entire modal-library release-cadence dependency and tightens CSP (no remote AppKit assets/analytics).
Axiswallet / connection
BasisdirectDESIGN.md sections 1, 4, 9: "No Drop Shadows," "Cards: no border-radius (or maximum 2px), sharp edges, 1px graphite borders," "Modals are sharp-cornered carbon panels with a 1px graphite border, centered on an obsidian scrim. No blur, no shadow." web/public/_headers:2 CSP currently allows https://*.reown.com and https://*.walletconnect.com for script/frame/connect sources — AppKit loads remote assets. externalWeb research: "a viem/wagmi-native custom connect UI sidesteps the modal-library lag entirely and plays better with strict CSP."
RationaleAppKit's default modal is rounded, shadowed, and themed for mass market — the opposite of DESIGN.md. Heavy theming overrides are possible but fragile across AppKit releases. A custom modal is more code to own but it is small, matches the design system by construction, and removes a third-party release-cadence dependency. This is a contingency for when AppKit theming proves heavier than expected, not a primary direction.
DownsidesWalletConnect QR rendering needs a library (qr-code-styling or similar) or canvas work — the one non-trivial piece. Loses AppKit's analytics/social logins (likely a pro for this team). More modal code to maintain. Only worth it if AppKit theming fights back; if AppKit bends cleanly to DESIGN.md, keep AppKit.
6

Promote Playwright + Anvil + mock EIP-1193 E2E from stretch goal to release gate

Conf MediumCx Medium-High
DescriptionThe plan's P5 makes a Playwright script of the 8-step two-wallet flow "a stretch goal, not a gate." Promote it: use the rombrom.com pattern (Playwright + Anvil mainnet fork + an injected mock EIP-1193 provider instead of a real MetaMask extension) to encode the manual checklist as a deterministic CI gate. Tenderly Virtual TestNets are the CI fallback if Anvil-in-CI proves flaky.
Axistesting-stack
BasisexternalWeb research: "The observed 2025-2026 pattern is two-tier: Vitest + mock connector for logic, and Playwright-based E2E for release gating. A leaner alternative avoids the [MetaMask] extension entirely: Playwright + Anvil + an injected mock EIP-1193 provider — deterministic and CI-friendly, matches your local-Anvil-fork dev loop." Tenderly Virtual TestNets (docs current Jul 2026) are positioned for CI/staging forks. directThe plan's P5 gate is currently "all 8 steps + npm run lint:security && test && build" where the 8 steps are a manual checklist; KTD9 specifies provider-tree integration tests for write flows but does not gate on a full E2E flow.
RationaleFor a DeFi protocol handling real value, a manual checklist as the release gate is a judgment call worth interrogating. The mock-EIP-1193 pattern is deterministic, reuses the existing local Anvil fork, and catches integration drift (ABI rename, CSP regression, contract/event mismatch) that unit tests miss. The existing web/tests/components/ModalErrorBoundary.test.tsx and provider-tree tests are the unit tier; this closes the E2E tier.
DownsidesAnvil-in-CI can be flaky (Tenderly VTN mitigates). More CI setup and maintenance. The manual 8-step flow is already documented and works for a small team. Whether this is "best" depends on release cadence and team size — surface it as a bar-raising option, not a forced upgrade.
