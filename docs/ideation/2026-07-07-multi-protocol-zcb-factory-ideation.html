Ideation: Multi-Protocol ZCB Factory Modularization

How to support multiple zero-coupon bond protocols (Pendle, Frax Bonds, Notional fCash) beyond the current Pendle-only architecture, and whether a modular factory or a wrapper/adapter approach is the right path.

2026-07-07 topic: multi-protocol-zcb-factory repo-grounded

Grounding Context

Codebase Context

OVRFLO is a Foundry/Solidity vault system for yield tokenization, currently hard-coupled to Pendle Finance. Users deposit Pendle Principal Tokens (PTs), receive ovrfloTokens (immediate portion) + a Sablier V2 stream (discount portion), and claim PT 1:1 after maturity. A secondary market (OVRFLOBook) allows selling/lending against streams. The factory (OVRFLOFactory) is owned by a timelocked multisig and is the single admin hub.

The Pendle coupling is concentrated in 4 call sites: IPendleOracle.getOracleState (freshness), IPendleOracle.getPtToSyRate (pricing, at 5 locations in the vault), IPendleMarket.readTokens (SY/PT/YT introspection), and IStandardizedYield.yieldToken (underlying validation). The oracle is an immutable singleton at both factory and vault level. The claim path (IERC20(ptToken).safeTransfer) and StreamPricing's marketActive / requireEligible are already protocol-agnostic. The registry interfaces (IOVRFLOSeriesRegistry) leak two Pendle fields: twapDurationFixed and oracle.

External Prior Art

Frax Bonds (FXB): ERC20 ZCB, 1:1 to FRAX at maturity, GDA pricing, per-series oracle, single underlying. Notional fCash: ERC1155, internal AMM pricing, multiple currencies/decimals. wrapped-fcash (closest prior art): ERC20+ERC4626 wrapper per (currencyId, maturity) via CREATE2+BeaconProxy. Yearn V3: ERC4626 tokenized strategies, thin 3-function adapter. Morpho V2: allocate/deallocate adapters, timelocked, per-adapter caps, frontrun risk on removal. ERC4626 is the convergent adapter seam. Survivor bias: Pendle dominates; Element and Yield Protocol both exited.

Institutional Learnings

AGENTS.md: "Keep code Pendle-specific... use a wrapper/adapter contract if another protocol is ever needed." Pattern #9: factory owns every dependent; admin forwarded. Pattern #10: one vault per underlying; cross-market ovrfloToken fungibility is a design feature. R-05: protocol-level redemption rejected (SY asset != yield token breaks 1:1 invariant); decision was "keep vault Pendle-specific and thin, push variability to outer layer." StreamPricing + registry interfaces are the existing abstraction seam.

Topic Axes

1. Protocol abstraction surface 2. Factory deployment & series routing 3. Oracle/pricing normalization 4. Vault-level decoupling 5. Claim/redemption path

1.

Adapter-per-series, vault-per-underlying: the USB class-driver model

Axis 2: Factory & routing Conf 88% Cx Medium-High
Description
The answer to "modular factory vs wrapper/adapter" is neither in its pure form. It's a third option modeled on USB device classes: one vault (class driver) per underlying, many adapters (devices) feeding series into it. The factory routes adapters to vaults by underlying via adapter.underlying(), preserving pattern #10's one-vault-per-underlying and cross-market ovrfloToken fungibility. A Pendle wstETH adapter and a Notional wstETH adapter both feed series into the same wstETH vault. addMarket collapses to addSeries(ovrflo, adapter, feeBps) — the adapter supplies pt/maturity/oracle-internals.
USB class model: one vault (class driver) per underlying, many adapters (devices) feeding series. Fungibility preserved within each vault.
Basis
direct: OVRFLOFactory.sol mapping(address => address) public underlyingToOvrflo (1:1, prevents duplicates); AGENTS.md pattern #10: "cross-market ovrfloToken fungibility under one underlying is a design feature." external: USB 2.0 spec class drivers: one class driver handles many devices of the same class. reasoned: if PTs from different protocols both redeem 1:1 to the same underlying at maturity, their ovrfloTokens are economically fungible regardless of originating protocol.
Rationale
This is the non-obvious resolution to the modular-factory-vs-adapter question. A modular factory that creates protocol-specific vaults breaks pattern #10 (fragmented liquidity). A wrapper external to the vault doesn't solve the oracle singleton. The USB model preserves the fungibility feature, the admin-forwarding pattern (#9), and the 1:1 invariant while making the vault protocol-agnostic. Each new protocol adds an adapter, not a vault.
Downsides
Requires vault restructuring (SeriesInfo gains an adapter field, loses twapDurationFixed). The factory must validate adapter-underlying matching. If the uniqueness key stays underlying (see idea #5), different protocols' PTs commingle in one vault's claim path, which works only if every adapter's PT is a transferable ERC20.
2.

Ship the ~5-function IZcbSource interface — the seam already exists

Axis 1: Abstraction surface Conf 90% Cx Medium
Description
The vault's entire Pendle surface reduces to one scalar (rateE18) fed into pure math (_computeSplit), plus a freshness gate and three onboarding fields. The coupling table's "Protocol-Agnostic Equivalent" column is literally the spec: principalToken(), maturity(), ptRate() / convertToUnderlying(ptAmount), isPriceFresh(), underlying(). The adapter is a ~40-line renaming of call paths that already exist. The vault is already 60% protocol-agnostic (claim + StreamPricing are generic). The OVRFLOBook inherits multi-protocol for free once two registry leaks (twapDurationFixed, oracle) are cleaned. Aligning with ERC4626 means wrapped-fcash IS already the Notional adapter — zero new code for that protocol.
Before: 4 Pendle interfaces across factory + vault. After: 1 IZcbSource interface, protocol-specifics inside adapters.
Basis
direct: OVRFLO.sol _computeSplit(uint256 ptAmount, uint256 rateE18) is pure math; claim does IERC20(ptToken).safeTransfer (generic, no Pendle); StreamPricing.sol marketActive calls only registry interfaces (no Pendle). external: wrapped-fcash is already an ERC20+ERC4626 wrapper for Notional fCash; Yearn V3 uses a thin 3-function adapter pattern. reasoned: the "deliberately lossy" LLVM-IR principle: the interface captures only the common subset (5 functions), preventing bloat when someone wants to add protocol-specific features.
Rationale
Reframes the decision from "should we do a big modular refactor?" to "should we extract 4 calls into a named interface?" The cost is small and one-time; the payoff is permanent protocol extensibility. The risk is much lower than it appears — the agnostic boundary already exists, it just hasn't been formalized. The half-abstracted registry interfaces (Pendle fields in "generic" interfaces) are technical debt that should be resolved one way or the other.
Downsides
The registry interface change (IOVRFLOSeriesRegistry.series() drops twapDurationFixed + oracle) ripples into StreamPricing and the OVRFLOBook — a 3-contract interface refactor. The "deliberately lossy" discipline must be enforced: every time someone wants a 6th function "just for Notional," the answer is no — that stays inside the adapter.
3.

Per-series adapter replaces the immutable oracle singleton

Axis 3/4: Oracle & vault decoupling Conf 85% Cx Medium
Description
The oracle is address public immutable oracle at both the factory (singleton for all markets) and the vault (constructor-set, constant per vault). This is the deepest coupling — once a vault is deployed, it can never use a different price source, making a non-Pendle series literally impossible on an existing vault. The fix: store an adapter address per-series in SeriesInfo; deposit/flashLoan call adapter.ptRate() + adapter.isPriceFresh() instead of the vault-level oracle. Remove the vault oracle immutable, remove _requireOracleFresh, remove twapDurationFixed from the struct. Immutability migrates from "Pendle oracle address" to "which adapter" — the same security property (no post-deposit price-source change) at finer granularity.
Basis
direct: OVRFLOFactory.sol:62 address public immutable oracle; ("Pendle TWAP oracle address, singleton, same for all markets"); OVRFLO.sol:51 address public immutable oracle; ("constant per vault"); OVRFLO.sol _requireOracleFresh calls IPendleOracle(oracle).getOracleState (Pendle-specific); SeriesInfo.twapDurationFixed is a Pendle TWAP window stored per-series. reasoned: the security property is "pricing logic can't be swapped post-deploy" — an immutable adapter reference per series satisfies this identically while allowing different series in the same vault to use different price sources.
Rationale
This is the prerequisite for everything else. Without breaking the oracle singleton, no other idea works — a vault deployed with Pendle's TWAP oracle can never host an FXB GDA series or a Notional AMM series. The rate call appears at 5 sites in the vault (deposit, flashLoan, preview functions), each paired with a freshness check — 10 places to keep consistent per protocol. An adapter-owned ptRate() + isPriceFresh() collapses all 10 to 2 single calls and guarantees consistency by construction.
Downsides
Removing the vault-level oracle immutable is a constructor signature change affecting deployment scripts and the factory's deploy(). Each adapter must be independently audited for price manipulation resistance. The prepareOracle ceremony (Pendle cardinality provisioning) moves into adapter.prepare() — a no-op for non-TWAP protocols but a new interface function.
4.

R-05 reframed: the invariant is "ERC20-ZCB-with-fixed-maturity," not "Pendle PT"

Axis 5: Claim & redemption Conf 82% Cx Medium
Description
R-05 rejected protocol-level redemption because Pendle's SY accounting-asset (stETH) != yield token (wstETH), breaking the 1:1 invariant. But that mismatch is Pendle-specific, not universal. The real invariant is: the vault transfers an ERC20 ZCB token 1:1, it never redeems. This forbids the R-05 mistake by construction while permitting multi-protocol. Adapters declare a redemption capability: canRedeemToUnderlying() returns false for Pendle (user claims raw PT, redeems on Pendle themselves), true for FXB (clean 1:1 to FRAX). R-05 becomes a per-adapter guardrail, not a global wall. Notional fCash (ERC1155) requires the wrapped-fcash ERC20 wrapper as its adapter — the adapter IS the token-shape normalizer.
Basis
direct: OVRFLO.sol claim: OVRFLOToken(ovrfloToken).burn(msg.sender, amount); IERC20(ptToken).safeTransfer(msg.sender, amount); — hardcoded 1:1 ERC20 transfer, no Pendle call, no redemption. R-05 decision doc: "Do not add a protocol-level redemption path... build it as an off-chain or zap-layer concern, never as a vault accounting change." external: FXB is 1:1 redeemable to FRAX at maturity; wrapped-fcash mints ERC20 1:1 against fCash. reasoned: R-05's concern is SY-logic contamination of the vault; an invariant defined as "transfer the ZCB, never redeem" forbids that for all protocols, not just Pendle.
Rationale
This is the idea that unblocks the whole direction despite the project's documented resistance to generalization (R-01, R-05). It doesn't reverse R-05 — it generalizes the invariant R-05 was protecting. The vault stays "thin" (R-05's prescription), but "thin" now means "transfer-only, protocol-agnostic" instead of "Pendle-specific." The adapter is the outer layer R-05 said to push variability to. The capability descriptor is the non-obvious edge-case guardrail: the happy path (deposit) is uniform, but the error/edge path (redemption) leaks protocol identity if the interface doesn't surface it.
Downsides
Adding an opt-in adapter-mediated redemption path (for FXB) revisits R-05 and needs explicit team buy-in. The capability descriptor adds interface complexity (a bool + a conditional code path). If canRedeemToUnderlying() is ever true for a protocol where the 1:1 doesn't hold, the invariant breaks — the adapter must be trusted on this claim. Raw Notional fCash (ERC1155) cannot use the current claim path at all; the ERC20 wrapper is mandatory, adding a dependency.
5.

The uniqueness key is an irreversible product decision: fungibility vs isolation

Axis 2: Factory & routing Conf 80% Cx Low (decision)
Description
Pattern #10 keys vaults by underlying alone (underlyingToOvrflo, 1:1, prevents duplicates). The moment a second protocol is added on the same underlying, the key collides: a Pendle wstETH vault and a Notional wstETH vault cannot coexist. Two options, no free lunch: (A) Keep key as underlying — cross-protocol ovrfloToken fungibility, deeper liquidity, multi-protocol as a unified ZCB aggregator, but commingles PTs with different redemption mechanics in one vault. (B) Promote key to (underlying, adapter) — isolation per protocol, pattern #10 preserved per-adapter, but ovrfloToken fungibility is lost across protocols (a Pendle-wstETH ovrfloToken != a Notional-wstETH ovrfloToken). This is a product-level call, not a refactor.
Basis
direct: OVRFLOFactory.sol mapping(address => address) public underlyingToOvrflo; ("Maps an underlying asset to its deployed OVRFLO vault, 1:1, prevents duplicates"); AGENTS.md: "cross-market ovrfloToken fungibility under one underlying is a design feature, not a bug." reasoned: fungibility requires identical redemption mechanics; two protocols over the same underlying have different mechanics (Pendle SY vs FXB direct vs Notional AMM settlement), so the fungibility boundary must track either the asset (option A, commingled) or the adapter (option B, isolated).
Rationale
This is the one place where the modular-factory-vs-adapter choice forces an irreversible product-level call. Option A turns OVRFLO from a Pendle wrapper into a unified ZCB yield aggregator — each new protocol deepens liquidity rather than fragmenting it, creating a network effect. But it requires trusting that all adapters' PTs are ERC20-ZCB-shaped (idea #4's invariant). Option B is safer but sacrifices the liquidity-aggregation thesis. The team must decide before the first adapter is built, because it determines deployment topology, token fungibility, and the product narrative.
Downsides
Option A's commingling risk: if one protocol's PT has a non-1:1 redemption (like Pendle's stETH/wstETH gap), it pollutes the fungibility of the shared ovrfloToken. Option B's fragmentation: loses the "unified liquidity" value proposition that might be the main reason to go multi-protocol. There is no hybrid that preserves both fungibility and isolation.
6.

Adapters make multi-protocol rationally attemptable: marginal, retractable, bounded-risk

Axis 2: Factory & routing Conf 85% Cx Low (argument)
Description
Today, supporting protocol N is a binary lumpy cost: factory changes, vault changes, oracle integration, full re-audit — all before a single depositor. Element Finance and Yield Protocol both exited. If the onboarded protocol dies in 6 months, all that work is stranded and the audit spend is sunk. An adapter model converts this to marginal, retractable cost: ship a thin adapter, measure demand, delist if the protocol dies with zero vault/factory changes. The adapter's blast radius is split-only — claim is a hardcoded burn(amount) + safeTransfer(amount) that never calls the adapter, so a malicious adapter can mis-split the deposit (rate manipulation) but cannot steal principal. Audit economics flip: 1 vault + N thin adapters is cheaper to audit than N protocol-specific vaults.
Basis
direct: OVRFLO.sol claim: burn(msg.sender, amount) + IERC20(ptToken).safeTransfer(msg.sender, amount) — no adapter/oracle call; _computeSplit: toUser + toStream = ptAmount — the rate only affects the split, not the total minted. external: Element and Yield Protocol both exited (observed protocol mortality); Morpho V2 uses per-adapter caps with timelocked removal. reasoned: if claim never consults the adapter, the adapter's blast radius is the split, not principal custody — a much lower trust bar that could even be insured/contested off-chain.
Rationale
This is the dominant economic argument and it's invisible in pure code analysis. For a multisig-governed project that moves slowly and re-audits expensively, retractability is the feature that makes multi-protocol rationally attemptable at all. Without it, the team would be betting a flagship-grade audit on each protocol's survival. With adapters, each protocol bears its own survival risk, scoped to its own adapter audit. The bounded-risk analysis (split-only, not principal) is the prerequisite for eventually allowing permissionless adapter onboarding.
Downsides
A malicious adapter can still grief depositors by returning a bad rate (over-splitting to immediate mint, depriving the stream; or under-splitting, locking too much in the stream). Per-series deposit caps (already a pattern) bound this. The "split-only" blast radius assumes claim stays hardcoded — if idea #4's adapter-mediated redemption is added, the blast radius expands to principal, raising the trust requirement.
7.

Duplication may beat abstraction at N≤3: force the crossover analysis

Axis 4: Vault decoupling Conf 75% Cx Low (analysis)
Description
The honest counterargument to the adapter direction. For N=2-3 protocols, full duplication (a vault copy per protocol, protocol-optimally simple, no adapter indirection) may be cheaper and clearer than a shared agnostic vault + adapters. Duplication cost is O(N x vault_size); adapter cost is O(interface_fixed + N x adapter_size). A crossover exists and is small. AGENTS.md's "favor simplicity and minimal abstractions" and "this is solidity not python" arguably favor duplication at small N. The forcing question: what is the crossover N? If the honest answer is "N>=4 and we'll have 3," duplication wins. If "we expect 5+ eventually," the adapter investment pays off.
Basis
direct: AGENTS.md: "Favor simplicity and minimal abstractions; pushback on added complexity ('this is solidity not python')"; "Keep code Pendle-specific... use a wrapper/adapter contract if another protocol is ever needed" — contemplates the adapter as a contingency, consistent with "duplicate first, adapter if it recurs." reasoned: duplication cost is O(N x ~600 lines); adapter cost is O(interface design + indirection gas + N x ~40 lines); crossover exists and is small (likely N=3-4).
Rationale
This is the strongest steelman of the "don't abstract" position and the team should hear it before defaulting to adapters. It gives AGENTS.md's simplicity preference a concrete architectural expression and forces a quantitative crossover analysis instead of a vibes-level "modular is good." Even if the team chooses adapters, this idea ensures the choice is deliberate and the crossover N is known, not assumed. It also frames the adapter work as an investment with a break-even point, not a free good.
Downsides
Duplication means N vault bytecodes to audit, N deployment flows, N admin surfaces — the audit-economics argument (idea #6) cuts against it. If the team duplicates at N=3 and then needs N=5, the migration from duplicated vaults to a shared+adapter vault is more painful than starting with adapters. The crossover analysis itself is speculative (protocol count is a product forecast, not a technical measurement).
